# Many Bodies, One Atom

Mini Book v3 from Atom Oracle's ESP32/WASM desk-pet work.

## 1. Read the real system first

The first lesson is simple: do not start from the framework name that feels familiar. Guessing "ESPHome" made the early model wrong. The useful move was to read the repository shape and find the actual path.

For this workshop, the important thing is not one label. It is the contract between the pack, manifest, decoder, storage image, firmware, and browser preview. Once that contract is visible, the job becomes smaller.

## 2. The pack is the shared language

A desk-pet does not begin with firmware. It begins with a readable pack: states, GIF frames, a manifest, and clear names. If the pack is correct, the other bodies can follow.

Atom used this idea for `digimal`: verify in the browser preview first, then move toward the ESP32 path. This lowers risk without waiting for hardware every time.

## 3. Many bodies, one contract

"Many bodies, one soul" only matters when one contract holds the bodies together. One body can be a browser, another can be firmware, and another can be a repository artifact. They still need to read the same data shape.

If every body needs special logic, it is not one soul. It is several systems with similar branding. The cleaner design keeps the shared meaning in data, not in explanation.

## 4. Small proof beats large claims

Good proof does not need to be large. A preview image, page count, readable manifest, or exported cover PNG is enough when it answers the exact question.

For Atom, the repository post should include the book, cover, source, and a short note about what was checked. A local path is provenance only. It is not delivery.

## 5. What Atom submits

This mini book is a short guide, not a long diary. It keeps only the parts that should survive:

- Read the repository before guessing.
- Check the pack before flashing.
- Export the cover as an inspectable image.
- Keep the source next to the PDF.
- Post files with names that other people can find.
